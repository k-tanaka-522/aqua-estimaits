🛠️ 効果的だったツールと手法

1. HTMLワイヤーフレーム

⭐⭐⭐⭐⭐

アプローチ: シンプルなHTML + CSSで画面イメージを作成

メリット:

  • ✅ ブラウザで即座に確認可能
  • ✅ レスポンシブデザインの検証が簡単
  • ✅ 実装イメージが具体的になる
  • ✅ クライアントとの認識合わせに有効

成果物: 6つのHTMLワイヤーフレーム（スマホ・PC版）

2. Draw.io XML図

⭐⭐⭐⭐⭐

アプローチ: ERダイアグラム、画面遷移図をXML形式で作成

メリット:

  • ✅ Git管理可能（バイナリではない）
  • ✅ VSCodeで編集・プレビュー可能
  • ✅ 変更履歴が追跡できる
  • ✅ 無料で高機能
# VSCode拡張機能をインストール "Draw.io Integration" # .drawioファイルをクリックで編集可能 database-er-diagram.drawio screen-transition-diagram.drawio

3. モックデータ駆動開発

⭐⭐⭐⭐⭐

戦略: 「張りぼてのフロント」優先アプローチ

メリット:

  • ✅ UIの早期確認
  • ✅ クライアントフィードバックが早い
  • ✅ フロント・バックエンド並行開発可能
  • ✅ デザイン変更のコストが低い

💡 学んだベストプラクティス

1. プロトタイプファースト

教訓: モックデータで先にUIを作ると、クライアントフィードバックが早く得られる

次回への適用: 全プロジェクトでモックデータアプローチを標準化

2. Git管理可能な形式を選ぶ

教訓: HTMLワイヤーフレーム、Draw.io XMLならGitで変更履歴が追える

次回への適用: バイナリ形式（.psd, .sketch）を避け、テキストベース・XML形式を優先

3. 段階的実装（SQLite → PostgreSQL）

教訓: シンプルから始めて後で切り替え

次回への適用: 開発環境は軽量に、本番環境は堅牢に、抽象化レイヤーで切替容易に

4. ドキュメント自動生成（Swagger）

教訓: 手動ドキュメント不要

次回への適用: すべてのAPIプロジェクトにSwagger導入

🔌 実装済みAPIエンドポイント

Lands Controller

GET /api/lands - 全土地取得 GET /api/lands/{id} - 特定の土地取得 POST /api/lands - 土地登録 PUT /api/lands/{id} - 土地更新 DELETE /api/lands/{id} - 土地削除

Simulations Controller

GET /api/simulations?landId={landId} - シミュレーション一覧取得 GET /api/simulations/{id} - 特定のシミュレーション取得 POST /api/simulations - シミュレーション登録 PUT /api/simulations/{id} - シミュレーション更新 DELETE /api/simulations/{id} - シミュレーション削除
